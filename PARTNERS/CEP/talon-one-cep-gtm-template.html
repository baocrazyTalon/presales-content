The Value Narrative

Why Talon.One + [CEP Name] is greater than the sum of its parts

Technical Synergy

The Promotion Brain & The Communication Voice

Most CEPs include a basic voucher or discount field. What they cannot natively do is validate, govern, personalise, and track those incentives with the rigour required to protect margin. Talon.One fills this gap precisely — and the CEP handles what it does best: orchestration and delivery.

🧠 Talon.One — The Promotion Brain

  • Complex rule logic (ANY / ALL / NOT conditions across any attribute)
  • Coupon validation: one-time, per-customer, per-segment, channel-scoped
  • Fraud prevention: domain checks, velocity limits, device fingerprinting
  • Loyalty ledger: earn, burn, tier evaluation, expiry
  • Referral tracking: cross-session attribution, reward triggers
  • Gamification: streaks, missions, stamp cards, challenges
  • Full audit trail: every decision logged per customer per session
  • Budget caps: spend limits, redemption ceilings, campaign-level guards

📣 [CEP Name] — The Communication Voice

  • Cross-channel orchestration: push, email, SMS, in-app, web
  • Customer segmentation: behavioural, predictive, RFM-based
  • Journey automation: event-triggered flows, branching logic
  • Content personalisation: Liquid / dynamic content blocks
  • A/B and multivariate testing at message level
  • Frequency capping and send-time optimisation
  • Conversion tracking: open/click/purchase attribution
  • CDP data relay: profile enrichment from behavioural events
The Core Philosophy The CEP decides who gets a message, when, and on which channel. Talon.One decides what incentive they are entitled to and whether they are allowed to use it. Neither tool can fully replace the other — together they close the loop between personalised communication and validated, economically-controlled incentive execution.
The Logic Gap in Standard CEPs Most CEPs allow a marketer to insert a coupon code into a message template. They do not natively validate: whether the customer already used that code, whether the code is valid for the customer's segment, whether budget caps have been hit, or whether the code is being shared/misused. Every redemption without Talon.One validation is a promotion event the CEP cannot govern — leading to silent margin leakage.
Value Proposition

How the Integration Prevents Margin Erosion & Increases ROI

The combination addresses the two fundamental risks in promotional marketing: giving away discounts to customers who would have converted anyway, and allowing discount codes to be redeemed outside their intended scope.

🛡 Margin Protection (Stop Leakage)

  • Single-use codes: technically enforced, not policy-only
  • Per-customer redemption caps prevent bulk exploitation
  • Email domain validation prevents duplicate account arbitrage
  • Budget caps: a campaign stops issuing at £X spend, not £X+leak
  • Velocity rules: flag and block anomalous redemption rates
  • Channel scoping: in-store codes rejected online (and vice versa)

📈 ROI Amplification (Earn More)

  • Right person: CEP segments deliver to highest-propensity customers
  • Right offer: T1 calculates personalised discount depth per profile
  • Right time: CEP send-time optimisation maximises open rates
  • Right channel: CEP routes based on prior engagement pattern
  • Right message: Liquid-driven content adapts to loyalty tier, history
  • Incrementality: combined data enables "would have bought anyway" modelling
Discovery Question for Prospects "How many discount codes that you send via [CEP Name] today are redeemed per customer? How many have been shared outside your intended audience? Do you have a campaign-level spend cap that is technically enforced — or just a policy?" — The answer to any of these usually reveals immediate, quantifiable margin leakage that the integration directly solves.

Use Case Matrix

Category-by-category breakdown of what to build, how, and what data bridges the two platforms

Category 01

Customer Acquisition

Incentivise the first purchase and the first referral — with controls that prevent incentive abuse at the widest point of the funnel.

1.1

Personalised Referral Programme

Unique referrer codes, cross-session referee attribution, fraud-proof reward triggers
High Impact
Why Combined? The CEP handles the referral invite message (whichever channel the customer prefers) but cannot attribute the referee's purchase to the referrer across sessions — nor validate that the referee is genuinely new. Talon.One assigns the unique referral code, tracks cross-session attribution, validates that the referee email domain is new, and triggers the referrer reward only on confirmed first purchase.

🧠 Setup in Talon.One

  • Create a Referral Campaign with auto-generated unique codes per advocate (format: REF-{customerId})
  • Set validation: referee email not in existing customer DB, referee first purchase ≥ £X minimum
  • Set referrer reward: issue £Y coupon or N loyalty points on referee order close
  • Set referee reward: auto-apply Z% discount on first order
  • Add velocity guard: max 5 referral rewards per advocate per 30 days

📣 Setup in [CEP Name]

  • Add "Share your referral code" CTA in post-purchase journey step
  • Pull unique referral code from Talon.One custom attribute stored on customer profile
  • Render code via Liquid: {{customer.attributes.referral_code}}
  • On referee "Order Completed" event: trigger a "Thanks for joining" email to referee + "Your reward is ready" email to advocate
  • Segment: only send to customers with loyalty_status = active and orders ≥ 1

🔗 Data Bridge

  • T1 → CEP: Webhook couponCreated pushes referral code to CEP custom attribute on advocate profile
  • CEP → T1: POST /v2/customer/sessions on referee checkout with referral code in session attributes
  • T1 → CEP: Webhook referralRewardGranted triggers advocate reward notification journey
  • T1 → CEP: Custom attribute referral_count updated — CEP uses it to unlock milestone messaging ("You've referred 5 friends!")
📋 Presales Customisation Note Replace "£Y coupon" and "Z% discount" with the prospect's standard referral reward value. For B2B prospects, consider account-level referral attribution (company email domain matching) instead of individual email validation.
1.2

First-Purchase Welcome Offer with Fraud Guard

Generous new-customer discount that cannot be reused, shared, or arbitraged
High Impact
Why Combined? CEPs can send a welcome email with a discount code included as plain text. They cannot prevent the same customer from creating a second account to reuse it, prevent the code from being posted on coupon aggregator sites, or cap total spend against the campaign. Talon.One adds all three controls technically.

🧠 Setup in Talon.One

  • Create a Coupon Campaign: generate one unique code per new customer
  • Validation rules: Is First Purchase = true, code usage limit = 1, per-customer cap = 1
  • Add email domain check: block known disposable-email domains (Mailinator, Guerrilla Mail, etc.)
  • Set campaign budget cap: max total discount spend = £X before campaign pauses
  • Set referral exclusion: do not apply if referral code is also in session

📣 Setup in [CEP Name]

  • Trigger: Account Created or Email Verified event
  • Wait 1 hour (let T1 generate and sync the unique code)
  • Send Welcome email with personalised code from attribute {{customer.attributes.welcome_code}}
  • Day 3 reminder if no purchase: re-send with urgency copy ("Your code expires in 4 days")
  • Suppress: cancel reminder if first_purchase_completed = true

🔗 Data Bridge

  • CEP → T1: POST /v1/customers creates customer profile on Account Created event
  • T1 → CEP: Webhook couponCreated pushes welcome_code attribute to CEP profile
  • T1 → CEP: Webhook sessionClosed with effect redemption sets first_purchase_completed = true to suppress the reminder
1.3

Partner / Affiliate Co-Marketing Codes

Trackable, channel-scoped codes generated per partner with shared attribution reporting
Medium Impact
Why Combined? Partners want to see their unique code performance. CEPs can personalise messages using a partner-attributed code, but cannot enforce which partner's code is redeemable in which channel or report on partner-level redemption velocity. Talon.One adds per-partner attribution metadata and can lock codes to a specific channel (web only, app only).

🧠 Setup in Talon.One

  • Create one Coupon Campaign per partner with a shared code or batch of unique codes
  • Add custom attribute partner_id on each code for attribution tracking
  • Set channel scope: valid only if session.channel = partner_referral
  • Set expiry: code batch expires on agreed partner deal end date
  • Export redemption report by partner_id for partner billing/settlement

📣 Setup in [CEP Name]

  • Receive new lead from partner landing page — create profile with segment tag source_partner_X
  • Send onboarding email including partner-specific code pulled from T1 coupon batch
  • Track email click and land on partner co-branded experience
  • On purchase: fire "Order Completed" to T1 with session.channel = partner_referral

🔗 Data Bridge

  • T1 → CEP: Batch coupon export via GET /v1/campaigns/{id}/coupons — import as CEP content block variable per partner segment
  • CEP → T1: Session payload includes channelType: partner_referral and partnerId custom attribute on customer update
  • T1 → CEP: Webhook couponRedeemed updates partner attribution counter on lead profile
Category 02

Conversion & Revenue

Recover abandoned carts and uplift basket size with offers that are personalised by customer value — not broadcast identically across the full database.

2.1

Real-Time Cart Abandonment with Dynamic Discount

Discount depth automatically calibrated to customer tier — not a blanket 10% for everyone
High Impact
Why Combined? CEP cart abandonment flows today typically send the same discount to all abandoners — giving away margin to customers who were already going to convert, while under-incentivising high-value customers who need a stronger nudge. Talon.One calculates the correct discount depth per customer based on their LTV, tier, and cart value — the CEP delivers it on the right channel at the right time.

🧠 Setup in Talon.One

  • Rule: IF customer.tier = Gold AND cart.value ≥ £80 → 15% off
  • Rule: IF customer.tier = Silver AND cart.value ≥ £50 → 10% off
  • Rule: IF customer.isFirstPurchase = true AND cart.value ≥ £30 → £5 off
  • Rule: IF customer.predictedChurnScore ≥ 0.7 → 20% off (reclaim)
  • Set: code valid for 24 hours, single use, can only be used by the specific customer (linked to customerId)
  • Set: campaign budget cap to prevent over-redemption during peak traffic

📣 Setup in [CEP Name]

  • Trigger: Cart Abandoned event (after 30-min inactivity)
  • Call T1 API or listen for T1 webhook to receive the personalised code
  • Step 1 (30 min): Push notification — cart still waiting, no code (test willingness to convert without incentive)
  • Step 2 (2 hours): Email with personalised code: "Here's {{offer_value}} off your basket"
  • Step 3 (24 hours): Final SMS — code expires tonight
  • Suppress all steps on Purchase Completed event

🔗 Data Bridge

  • CEP → T1: PUT /v1/customer/sessions/{sessionId} — send cart state (items, values) to T1 on "Cart Abandoned" trigger
  • T1 → CEP: Webhook couponCreated with discount_value and code attributes — CEP uses these in the message Liquid template
  • T1 → CEP: Webhook sessionClosed → CEP suppresses remaining steps
  • T1 attribute on customer: last_abandonment_discount_sent — prevents sending another discount within 7 days
📋 Presales Customisation Note Replace the tier names and cart value thresholds with the prospect's existing loyalty tiers and AOV data. If the prospect does not have a loyalty programme yet, use RFM decile from the CEP as the segmentation variable passed to T1 as a custom attribute.
2.2

Real-Time "Spend More, Save More" Basket Nudge

In-session progress bar communicating personalised spend thresholds
High Impact
Why Combined? CEPs cannot evaluate live basket state. "Spend £10 more to unlock free delivery" requires the promotion engine to know the current cart value and calculate the gap to the next threshold in real time — Talon.One does this per session, per rule tier, and sends the gap value to the CEP to include in in-app or web messages.

🧠 Setup in Talon.One

  • Create tiered rules: £50 threshold → free delivery / £75 → 10% off / £100 → free gift
  • Return custom effect: nextTierGap = (threshold − cart.total) in session response
  • Return: nextTierReward = "free delivery" / "10% off" / "free gift"
  • Rule fires on every UpdateSession call as items are added — no polling required

📣 Setup in [CEP Name]

  • Read nextTierGap and nextTierReward from the T1 API response on each cart update
  • Render in-session in-app message: "Add {{nextTierGap}} more to get {{nextTierReward}}"
  • On threshold unlock: fire a celebratory banner — "You unlocked free delivery!"
  • Post-session: if customer left without reaching next tier, send an email with the gap and a time-limited reminder

🔗 Data Bridge

  • Storefront → T1: PUT /v1/customer/sessions/{id} on every cart item add/remove
  • T1 → Storefront: Session response includes effects array with custom nextTierGap effect
  • Storefront → CEP: Pass nextTierGap and nextTierReward as content event for CEP in-app message render
2.3

Post-Browse Personalised Retargeting Offer

Category-specific discount triggered by browsing intent signal from CEP, validated by T1
Medium Impact
Why Combined? The CEP captures the browse signal and knows which category the customer was exploring. Talon.One generates a targeted offer specific to that category — not a generic sitewide code — ensuring the offer is only redeemable on the category the customer was viewing, preventing the discount from being applied to higher-margin items.

🧠 Setup in Talon.One

  • Create campaign: IF session.attributes.browsed_category = X AND coupon.code is valid → apply Y% off category X items only
  • Generate unique code per browse event (linked to customer + category)
  • Rule: code only valid on items tagged with category = browsed_category
  • Expiry: 48 hours from code generation

📣 Setup in [CEP Name]

  • Trigger: Product Category Viewed ≥ 3 times without purchase in session
  • Call T1 to generate category-scoped coupon (pass browsed_category as session attribute)
  • Send email / push 2 hours after browse: "Still thinking about [Category]? Here's a code."
  • Suppress if purchase in that category occurred within the 2-hour window

🔗 Data Bridge

  • CEP → T1: POST /v1/customer/sessions with attributes.browsed_category to trigger coupon creation
  • T1 → CEP: Webhook couponCreated pushes category-scoped code to CEP for inclusion in message

Loyalty Milestone Reward

Automated reward + celebration message triggered the moment a customer hits a points/spend threshold

🔗 Data Bridge

  • T1 → CEP: Webhook sessionClosed with effect milestoneReached — CEP uses as journey trigger

3.2

Tier-Up Celebration & Onboarding

Multi-step welcome journey into a new loyalty tier, with tier-exclusive offer unlocked immediately
High Impact
Why Combined? Tier transitions are high-emotional-value moments. A transactional "you've been upgraded" system email wastes the moment. The CEP transforms the webhook into a multi-touch welcome journey; Talon.One simultaneously unlocks tier-exclusive offers and validates eligibility for new mechanics (e.g. early access, exclusive coupons) the customer couldn't access before.

🧠 Setup in Talon.One

  • Define tier thresholds in Loyalty programme (e.g. Bronze → Silver at 1,000 pts lifetime)
  • On tier change: issue tier-welcome coupon (unique, valid 30 days, Gold-only)
  • Fire effect tierChanged with old_tier, new_tier, welcome_offer_code
  • Update customer.tier attribute — all future rules now evaluate against new tier
  • Unlock access to tier-exclusive campaigns automatically via loyalty tier condition

📣 Setup in [CEP Name]

  • Trigger: T1 webhook tierChanged → "Tier Up Celebration" journey entry
  • Immediately: Push notification — "Welcome to Gold! 🎉"
  • Day 0: Email — new tier benefits explained + exclusive welcome offer code
  • Day 3: In-app message — "Try your first Gold benefit: early access sale starts Friday"
  • Day 14: Check points balance — if no purchase, send "Don't lose your status" nudge

🔗 Data Bridge

  • T1 → CEP: Webhook sessionClosed with effect tierChanged — CEP journey trigger + attribute update loyalty_tier = Gold
  • T1 → CEP: welcome_offer_code in webhook payload — used in email Liquid template
  • CEP segmentation: loyalty_tier attribute used to gate tier-exclusive campaigns, early-access invitations, and content blocks

Gamification: Stamp Card & Mission Progress Nudges

Category 04

Win-Back & Re-Engagement

Protect the loyalty programme's active base by identifying at-risk members and deploying the right incentive at the right intervention point — before churn is confirmed.

4.1

Churn Prediction + Personalised Win-Back Offer

CEP's churn score unlocks a T1-generated win-back offer calibrated to the customer's historical basket value
High Impact
Why Combined? The CEP's ML churn model identifies at-risk customers. But sending the same 10% discount to all at-risk customers over-spends on those who would have returned naturally (low-churn-score customers) and under-incentivises those who need a stronger nudge (high-score). Talon.One uses the churn score passed as a custom attribute to calculate the optimal offer depth per customer.

🧠 Setup in Talon.One

  • Rule: IF customer.churn_score BETWEEN 0.5–0.69 → 10% off next purchase
  • Rule: IF customer.churn_score BETWEEN 0.7–0.84 → 15% off + double points
  • Rule: IF customer.churn_score ≥ 0.85 → 25% off + free delivery
  • Set: code valid 7 days, single use, linked to customerId
  • Set: global cap — do not send win-back offer if one was sent in the last 60 days

📣 Setup in [CEP Name]

  • Segment: customers inactive for 45 days with churn score ≥ 0.5
  • Sync churn_score to T1 customer attribute via PUT /v1/customers/{id}
  • Call T1 to generate win-back coupon; wait for couponCreated webhook
  • Step 1 (Day 0): Personalised email — "We miss you, {{first_name}}"
  • Step 2 (Day 3): Push with offer code if email unopened
  • Step 3 (Day 6): Final SMS with code expiry urgency

🔗 Data Bridge

  • CEP → T1: PUT /v1/customers/{id} — sync churn_score, days_since_last_purchase as T1 custom attributes
  • CEP → T1: Trigger coupon generation via POST /v1/customer/sessions with win-back campaign code
  • T1 → CEP: Webhook couponCreated — code and offer value passed to CEP for message personalisation
  • T1 → CEP: Webhook sessionClosed on purchase — suppresses remaining steps, sets winback_converted = true
📋 Presales Customisation Note Not all CEP partners have native churn scoring. For MoEngage: use RFM score. For Bloomreach: use Luigis Box prediction score. For Braze: use Predictive Churn built-in. Pass whichever score is available as a normalised 0–1 float to T1 as churn_score via a daily batch or real-time event update.
4.2

Points Expiry Warning + Burn Nudge

Notify at-risk points about to expire, then guide redemption into a purchase
Medium Impact

🧠 Setup in Talon.One

📣 Setup in [CEP Name]

  • Step 1 (30 days out): Email — "You have {{expiring_points_value}} points expiring on {{expiry_date}}"
  • Step 2 (7 days out): Push — "Don't lose your rewards — redeem now"
  • Step 3 (2 days out): SMS — final reminder with deep link to redemption page
  • Step 4: If converted — "Reactivation bonus" message if eligible

🔗 Data Bridge

  • T1 → CEP: Webhook pointsAboutToExpire — CEP journey trigger with expiring_points_value and expiry_date
  • CEP → T1: PUT /v1/customer/sessions on checkout — T1 calculates available points and applies burn effect

Tier-Based Early Access Campaign

Gold-only sale access, technically enforced — not just communicated
High Impact
Why Combined? Communicating VIP early access via CEP without technical enforcement means any customer with the link can access the sale — eroding the exclusivity that makes the tier valuable. Talon.One enforces that only Gold customers can redeem the early-access campaign, even if the URL leaks.

📣 Setup in [CEP Name]

  • Send VIP early-access email with unique deep link including customerId parameter
  • In-app announcement on app open for Gold members during early-access window

🔗 Data Bridge

  • CEP → T1: Session payload includes customerId — T1 looks up tier in real time
  • T1 → CEP: Customer attribute loyalty_tier — CEP uses as segmentation condition for VIP communication

Ease of Setup

A 3-step integration workflow — not a 6-month dev project

Implementation Workflow

Connect → Configure → Campaign

The standard integration between Talon.One and any major CEP follows a predictable 3-step pattern that a joint implementation team can complete in 2–6 weeks, depending on use case complexity.

01
Connect

Platform-to-Platform Link

  • Configure T1 API credentials in CEP webhook settings
  • Set up outbound T1 webhooks pointing to CEP event ingestion endpoint
  • Map T1 customer profile attributes to CEP custom attributes schema
  • Test bidirectional event flow with a sandbox customer profile
  • Validate that customerId is consistent across both systems (single key)
02
Configure

Attribute & Event Schema

  • Define custom attributes to sync: loyalty_tier, points_balance, churn_score, coupon codes
  • Create T1 webhook listeners for each journey trigger event type
  • Set up CEP Liquid variables for T1 attribute values in message templates
  • Define session attribute schema (cart value, channel, campaign tags)
  • Configure T1 campaign budget caps and rate limits
03
Campaign

Launch First Use Case

  • Build the first campaign in T1 (recommend: Welcome Offer or Cart Abandonment)
  • Build the corresponding journey in CEP using T1 webhook as trigger
  • QA end-to-end with test customer profiles across all branches
  • Go live with a 10% rollout, monitor redemption rates and latency
  • Expand to next use case from the matrix above
Key Message for Prospects: This Is Configuration, Not Code Once the initial Connect step is complete (typically 3–5 days of backend work), every subsequent use case in the matrix above is built by a marketer in Talon.One's Campaign Manager and a marketer in the CEP's journey builder. No additional development sprint is required per use case. The CEP sends data; T1 responds with a decision. It's a platform pattern, not a custom integration.
Standard Integration Components Used Across All Use Cases
  • T1 Webhooks (outbound): sessionClosed, couponCreated, couponRedeemed, tierChanged, achievementCompleted, pointsAboutToExpire
  • T1 API endpoints (inbound): PUT /v1/customers/{id}, POST /v1/customer/sessions, PUT /v1/customer/sessions/{id}, GET /v1/campaigns/{id}/coupons
  • CEP custom attributes synced from T1: loyalty_tier, points_balance, active_coupon_code, offer_value, offer_expiry
  • CEP events used as T1 API triggers: Cart Abandoned, Order Completed, Profile Created, Browse Event

Browsing / Intent Evaluate pre-session intent signals passed as attributes. Pre-calculate eligible campaigns. Respond to segment tags from CEP. Capture browse events (category, product, frequency). Compute intent score. Sync segment tags to T1 customer profile. PUT /v1/customers/{id} with browsed_category, intent_score CEP-computed attributes.
Active Cart / Checkout Evaluate all active campaigns per session. Return: applied discounts, loyalty effects, coupon validity, next-tier gap, custom effects. Enforce fraud rules. Display real-time progress bar ("Spend £X more to unlock Y") using T1 session response effects. No logic lives in the CEP. PUT /v1/customer/sessions/{id} on every cart item update. T1 returns effects array — CEP reads nextTierGap for in-session UI.
Order Confirmation Close and commit session. Evaluate post-purchase effects: award points, issue bounceback coupon, update tier, trigger milestone. Fire all outbound webhooks. Listen for T1 webhooks on order close. Trigger: order confirmation + points earned message. Branch: tier-up journey if tierChanged. Suppress pending abandonment journeys. T1 webhooks: sessionClosed, tierChanged, milestoneReached, couponCreated (bounceback) → CEP journey triggers.
Post-Purchase (Day 2–7) Validate bounceback coupon eligibility on next visit. Track loyalty balance. Update expiry countdown. Send "Here's a reward for your purchase" (bounceback) message. Send review request (no T1 integration needed). Sync updated points_balance to CEP profile. T1 attribute points_balance synced after session close → CEP uses in message content and for segmentation rules.
Re-Engagement (Day 45+) Generate win-back coupon scaled to customer LTV / churn score attribute. Set strict redemption guard (60-day cooldown between offers). Identify at-risk segment via churn model. Sync churn_score to T1. Trigger win-back journey. Orchestrate email → push → SMS sequence. PUT /v1/customers/{id} with churn_score → T1 generates scaled coupon → webhook couponCreated → CEP sends with personalised offer value.
Points Expiry Warning Evaluate expiry schedule per customer. Fire pointsAboutToExpire webhook 30 days before expiry date with expiry metadata. Listen for T1 expiry webhook. Trigger 3-step reminder journey: email (Day -30) → push (Day -7) → SMS (Day -2). T1 webhook pointsAboutToExpire with expiring_points_value and expiry_date → CEP journey entry with those values as message variables.
Tier Upgrade Evaluate tier threshold on session close. Issue tier-exclusive welcome coupon. Update customer.tier attribute. Unlock tier-exclusive campaigns. Listen for tierChanged webhook. Trigger Tier-Up Celebration journey: push → email (Day 0) → education (Day 3) → status-maintenance nudge (Day 14). T1 webhook tierChanged with new_tier, welcome_offer_code → CEP journey trigger + loyalty_tier attribute update for all future segmentation.
Referral Programme Generate unique referral code per customer. Validate referee on redemption (new email, first purchase). Award both referrer and referee rewards. Track referral velocity. Insert referral code in post-purchase email. Send referee welcome flow on account creation with referral tag. Notify referrer on successful friend's purchase. T1 webhook couponCreated → CEP attribute referral_code. T1 webhook referralRewardGranted → CEP referrer notification journey.

Integration Readiness Checklist

Use this before a demo or proof-of-concept to qualify integration complexity and surface blockers early.

Technical Readiness

Prospect has a consistent customer ID shared across CEP and ecommerce/POS systems
CEP supports outbound webhooks or event API publishing (most do natively)
CEP supports inbound custom attributes on customer profiles
CEP supports custom event triggers for journeys (not only standard e-commerce events)
Checkout / POS system can call T1 UpdateSession API at cart evaluation
Backend can receive T1 webhook events and route to CEP or internal systems
T1 sandbox environment provisioned for CEP integration testing
CEP Liquid/template system supports dynamic variable rendering from custom attributes

Data / Business Readiness

Customer loyalty tier data is available in CEP or can be synced from T1 attribute
Prospect uses (or plans to use) segmentation as a campaign targeting input
Marketing team has authority to set promotion budget caps per campaign
Prospect can define maximum discount value / redemption count per campaign
Birthday / anniversary data is collected and available for personalisation
Cart abandonment events are already tracked (or can be added) in CEP
Prospect has or is planning a referral programme use case
Prospect has a loyalty programme (or is planning one) with tier mechanics
Common Blockers to Surface Early
  • No consistent customer ID: The integration depends on a single customerId that is the same in both T1 and the CEP. If the prospect is using different IDs in their ecommerce and CRM systems, data unification is a prerequisite — resolve via CDP or ID-stitching before this integration.
  • Session API not implemented: Some prospects only use T1 for coupons today. If they haven't implemented the Session API, the real-time basket mechanics (cart abandonment, spend-threshold nudge) require additional backend work — but the webhook-only use cases (milestones, referrals, point expiry) don't.
  • CEP event model too simple: Older CEP configurations may only support standard e-commerce events (Purchase, Email Open). Confirm the CEP can receive and react to T1 custom webhook events as journey triggers.
