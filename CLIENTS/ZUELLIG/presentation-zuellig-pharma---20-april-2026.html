01 — Architecture

MACH Architecture & API-First Approach

Talon.One is built as a MACH-compliant promotion engine — Microservices, API-first, Cloud-native, and Headless. It will power both the loyalty program (points, tiers) and all discounts & promotions, consolidating into a single, unified engine for Zuellig Pharma's B2B platform. Talon.One connects seamlessly to Zuellig Pharma's existing systems via API.

🔲

Microservices

Independent, loosely-coupled services that scale individually. Each domain (rules, sessions, profiles, loyalty) operates autonomously.

⚡

API-First

Every feature is exposed via a REST API. No UI dependency — your backend calls Talon.One directly. Full control, full flexibility.

☁️

Cloud-Native

SaaS platform deployed on Google Cloud Platform (GCP). Auto-scaling, high availability, managed infrastructure. APAC region for low latency across Zuellig Pharma's markets.

🔗

Headless

No frontend imposed. Talon.One is a pure backend engine — your B2B portal, mobile app, or ERP calls the API and renders the experience your way.

Why this matters for Zuellig Pharma: Your team manages a complex B2B distribution platform across multiple markets. Talon.One slots in as a backend API service — no iframes, no widgets, no frontend lock-in. Your developers call the Integration API from your existing services and control every aspect of the user experience.

03 — Loyalty Programs

Loyalty Programs

Talon.One supports Zuellig Pharma's B2B Loyalty Program through flexible, configurable loyalty ledgers that can track points, tiers, and rewards across all customer touchpoints and markets.

🏆

Points Ledger

Each customer profile maintains a real-time points balance. Points can be awarded on purchase, redeemed at checkout, or expired based on configurable rules — all without custom code.

Each country can operate its own dedicated ledger with localised timezone support — ensuring that point-earning windows, campaign cutoffs, and expiry deadlines align precisely with local time, not a centralised UTC clock. Point expiration policies are configured independently per country ledger, allowing PH, VN, SG, KH, KR, and MY to each enforce distinct rolling windows or fixed reset dates that reflect their market calendars and regulatory requirements.

🎖️

Tier Management

Define Bronze, Silver, and Gold tiers with automatic promotion and demotion logic based on rolling spend windows or point thresholds. Tier status unlocks exclusive campaigns and multipliers.

🌏

Multi-Market Support

Separate loyalty ledgers per market (PH, VN, SG, KH, KR, MY) ensure that point earning rates, redemption rules, and tier thresholds can be tailored to local market conditions and regulations.

🔄

Points Expiry

Configurable expiry policies — rolling 12-month windows or fixed year-end resets — keep the programme commercially sustainable and incentivise regular engagement from HCP customers.

Per-Country Loyalty Configuration

Each market application has its own fully independent loyalty setup — ledger, tiers, earn rules, expiration, and timezone — configured separately within the shared Talon.One environment.

04 — Identifiers

Identifiers — Your System of Record

Customer profiles and sessions are identified by an IntegrationID — a unique string provided by Zuellig Pharma, not generated by Talon.One.

👤

Customer Profile Identifier

Use your existing customer ID from your CRM or ERP. Example: ZP-USER-98421

This becomes the key to link loyalty points, tiers, and rewards to the right individual HCP or B2B user. In a B2B context, multiple users may order on behalf of the same clinic or account — see the card-based model below.

🛒

Session Identifier

Use your order or cart ID from EZRx. Example: ORDER-2026-06-8842

Ensures idempotency — the same session ID always returns the same promotion result, preventing double discounts.

🌏

Multi-Market Identifier Strategy

For multi-country deployments, prefix the Identifier with a market code. Example: PH-USER-98421 or VN-USER-00312

This keeps customer profiles cleanly separated across markets and simplifies reconciliation in your data warehouse.

B2B Multi-User Model — Card-Based Loyalty

In EZRx's B2B context, multiple HCPs or staff members may place orders on behalf of the same clinic, hospital, or pharmacy. Talon.One's Card-Based Loyalty feature is the recommended approach to handle this: a single loyalty card represents the account (company), and multiple individual user profiles can be linked to it.

🏥

Loyalty Card = The Account

Each clinic or pharmacy is assigned a Loyalty Card (e.g. ZP-CARD-C001). The card holds the shared points balance and tier status for the entire account — regardless of which staff member placed the order.

👥

Multiple Users Linked to One Card

Each individual user (HCP, pharmacist, procurement staff) has their own profile linked to the account's card via the Link customer to loyalty card endpoint. All their orders contribute to the same card balance.

🎯

Flexible Points Earning

Campaign rules can award points to the card (account level) on every session close, regardless of which linked user placed the order. Tier upgrades, milestone rewards, and coupons are all managed at the card level.

Controlling Point Redemption at the Individual Level

While points accumulate at the account (card) level, Zuellig Pharma may want to control who can redeem points and under what conditions. Talon.One's Rule Builder allows fine-grained redemption control based on customer profile attributes, audience membership, and redemption value thresholds.

🧑‍⚕️

Attribute-Based Redemption

Redemption rules can check individual user profile attributes before allowing points to be spent. For example, only users with account_type = "authorised_buyer" or role = "procurement" can redeem — blocking junior staff from consuming the shared balance.

👥

Audience-Based Redemption

Segment users into Talon.One Audiences (synced from your CRM) and restrict redemption to specific groups. For example, only users in the "Gold Account Buyers" audience can redeem points above 500 pts per order.

🔢

Conditional Redemption Thresholds

Add rule conditions to cap or gate redemptions. Example: redemption is only allowed if the user's requested redemption value is above a minimum (e.g. ≥ 100 pts) or below a maximum per transaction (e.g. ≤ 1,000 pts), preventing micro-redemptions and protecting budget.

05 — Customers Experience and API

Customers Experience and API

05a — Customer Registration

Customer Registration and Update

Update customer attributes in Talon.One to power segmentation and targeting within campaign rules. Profiles can be updated via the Session API (automatically) or the dedicated Profile endpoint.

You can also sync audience memberships in real-time — push your segment/audience IDs alongside profile attributes so Talon.One campaign rules can target customers based on your own segmentation logic (e.g. from your CDP, CRM, or Clevertap).

Endpoint

PUT /v2/customer_profiles/{integrationId}

Example — Updating Customer Attributes

// PUT /v2/customer_profiles/ZP-CUST-98421 { "attributes": { "hcp_specialty": "general_practice", "market": "PH", "account_type": "clinic", "tier": "Silver", "lifetime_orders": 47, "last_order_date": "2026-03-20T10:30:00Z", "preferred_categories": ["OTC", "Cardio"] } }

With Audience Sync: Include the audiencesChanges object to add or remove audience memberships in real-time. Talon.One will evaluate campaign rules based on these audience IDs — enabling targeted promotions for specific customer segments from your CDP, CRM, or Clevertap.

// PUT /v2/customer_profiles/ZP-CUST-98421 { "attributes": { "hcp_specialty": "general_practice", "market": "PH", "account_type": "clinic", "tier": "Silver", "lifetime_orders": 47, "last_order_date": "2026-03-20T10:30:00Z" }, "audiencesChanges": { "adds": [ 14 // Add to audience ID 14 (e.g. "PH Silver HCPs") ], "deletes": [ 8 // Remove from audience ID 8 (e.g. "New Customers") ] } }

Attributes Power Campaign Rules

Once attributes are synced, the marketing team can create rules such as:

  • "If hcp_specialty = general_practice" → 2× points on primary care products
  • "If market = PH" → Philippines-exclusive campaign access
  • "If lifetime_orders > 50" → VIP bonus points tier
  • "If account_type = hospital" → Hospital-tier pricing and promotions

Bulk Update — Multiple Customer Profiles

Update (or create) up to 1,000 customer profiles in a single request using the batch endpoint. Ideal for initial data migration, nightly syncs from your CRM or ERP, or bulk attribute updates.

The integrationId must be a stable identifier. Do not use an ID the customer can update themselves — use a database or ERP ID.

PUT /v2/customer_profiles
// Batch update up to 1,000 profiles { "customerProfiles": [ { "integrationId": "ZP-CUST-98421", "attributes": { "hcp_specialty": "general_practice", "market": "PH" } }, { "integrationId": "ZP-CUST-10234", "attributes": { "hcp_specialty": "cardiology", "market": "VN" } } // ... up to 1,000 profiles per request ] }

05b — Transaction

Transaction

The Session API is the heart of Talon.One integration. It represents a purchase transaction / cart / order on EZRx and evaluates all active campaign rules in real-time.

Endpoint

PUT /v2/customer_sessions/{customerSessionId}

Session States

🟢 Open Cart is active, customer is ordering
→
🔵 Closed Order confirmed / paid
or →
🔴 Cancelled Order cancelled / refunded
StateWhen to useWhat happens in Talon.OneLoyalty Impact
open Cart updated, item added/removed, coupon applied Campaigns evaluated, effects returned (discounts, points preview, tier check) Points are pending — shown to customer but not yet credited
closed Order placed & payment confirmed Session locked, effects finalized, loyalty balance updated Points are credited to customer's loyalty ledger. Spend counted toward tier progression.
cancelled Order cancelled or fully refunded All effects rolled back automatically Points are reversed. Spend removed from tier calculation.

Example — Open Session (Cart Update)

// PUT /v2/customer_sessions/ORDER-2026-06-8842 { "customerSession": { "profileId": "ZP-CUST-98421", "state": "open", "cartItems": [ { "name": "Panadol 500mg x20", "sku": "PAN-500-20", "quantity": 10, "price": 8.50, "position": 0, "category": "OTC" }, { "name": "Voltaren Gel 100g", "sku": "VOL-GEL-100", "quantity": 5, "price": 14.90, "position": 1, "category": "Pain" } ], "couponCodes": ["WELCOME10"], "total": 159.50 } }

With Product Catalog Sync: When your product catalog is synced to Talon.One, cart items only need the sku, quantity, and position. Price, name, and other attributes are automatically read from the catalog.

// PUT /v2/customer_sessions/ORDER-2026-06-8842 { "customerSession": { "profileId": "ZP-CUST-98421", "state": "open", "cartItems": [ { "sku": "PAN-500-20", "quantity": 10, "position": 0 }, { "sku": "VOL-GEL-100", "quantity": 5, "position": 1 } ], "couponCodes": ["WELCOME10"], "total": 159.50 } }

Response — Effects for Zuellig Pharma to Render

// Talon.One returns effects to apply { "effects": [ { "effectType": "setDiscount", "props": { "name": "Coupon Discount", "value": 15.95, "position": 0 } }, { "effectType": "setDiscountPerItem", "props": { "name": "10% off OTC", "value": 8.50, "position": 0 } }, { "effectType": "addLoyaltyPoints", "props": { "name": "Points Earned", "programId": 12, "value": 320, "subLedgerId": "PH" } } ] }

Loyalty Flow: When a session is open, Talon.One returns a points preview (e.g., "320 points on this order"). When the session is closed, points are credited to the customer's loyalty ledger. If the session is cancelled, points are automatically reversed.

08 — Security

Security

Talon.One is built to meet the security requirements of regulated industries including healthcare distribution. Key controls relevant to Zuellig Pharma's compliance posture are outlined below.

🔒

TLS 1.2+ Everywhere

All API traffic is encrypted in transit using TLS 1.2 or higher. Talon.One does not support unencrypted HTTP connections to any endpoint.

🗝️

API Key Rotation

Integration and Management keys can be rotated at any time without downtime. ZP's secrets manager should hold keys and rotate them on a 90-day cycle.

👤

Role-Based Access

Talon.One's UI supports granular RBAC — marketing users can manage campaigns without accessing API credentials or financial reporting.

📋

Audit Logs

All Management API calls and UI actions are logged with user identity and timestamp, supporting SOC 2 and internal audit requirements.

🛡️

Data Residency

Talon.One offers regional deployments (EU, US, APAC). For Zuellig Pharma, the APAC region ensures customer data remains within the Asia-Pacific boundary.

✅

SOC 2 Type II

Talon.One holds SOC 2 Type II certification. The latest report is available under NDA for Zuellig Pharma's security review process.
