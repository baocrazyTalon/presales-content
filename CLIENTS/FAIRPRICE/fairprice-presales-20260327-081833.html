01 Architecture

A Composable Component for Your Modern Stack

Fairprice has invested in a best-of-breed, composable stack with Segment and Braze. The promotions engine is the critical third pillar that executes business logic. However, legacy platforms like Comarch and Capillary introduce architectural flaws that create bottlenecks and complexity.

The Legacy Flaw: Logic Silos

Competitors like Comarch and Capillary use monolithic architectures with separate, bolted-on modules for promotions, loyalty, and referrals. This creates 'logic silos' where rules can conflict, data is inconsistent, and omnichannel campaigns require complex orchestration. Their high-latency, legacy APIs are not designed for real-time data streams from modern CDPs like Segment, creating a performance bottleneck.

The Talon.One Advantage: Unified Engine

Talon.One is built on a MACH (Microservices-based, API-first, Cloud-native, Headless) architecture. Our Unified Rule Engine provides a single, consistent interface to build any rule, from a simple coupon to a complex loyalty program. A single rule can award loyalty points, apply a cart discount, and trigger a webhook to Braze simultaneously, eliminating the complexity and data sync issues of siloed systems.

02 API Integration Flows

Seamless Integration with Segment & Braze

Our architecture is designed for seamless data flow within your existing ecosystem, minimizing the level of effort for your team. Segment sends customer data and events to Talon.One; our rule engine evaluates this data in real-time; and we forward the outcomes ('effects') to Braze for personalized communication. This entire flow is orchestrated via our high-performance Integration API.

03 Environments

Structured Deployment from Dev to Production

Talon.One provides a structured environment setup to support your full development lifecycle. Each environment is isolated, ensuring that testing and development do not impact your live production campaigns. We call these environments "Applications" within the Talon.One Campaign Manager.

Development & QA

Isolated sandboxes for developers to integrate with our APIs and for QA teams to test campaign logic without using real data. Each Application has its own API keys.

Staging (UAT)

A pre-production environment that mirrors your production setup. Used for User Acceptance Testing (UAT) and final validation of campaigns and integration logic before go-live.

Production

The live environment handling all real customer traffic. We recommend separate Applications for different regions or business units to manage currencies, timezones, and data independently.

04 Key APIs

Engineered for Enterprise Performance & Scale

Talon.One offers two primary APIs, each designed for a specific purpose. All communication should happen from your backend to ours to avoid exposing sensitive credentials.

Integration API (Session V2)

This is the high-throughput, low-latency API for all real-time interactions. It's used to send customer session data and receive effects from the Rule Engine. We guarantee sub-50ms rule evaluation, critical for checkout and POS use cases.

curl --request PUT 'https://your_deployment.talon.one/v2/customer_sessions/cart-uuid-123' \
--header 'Authorization: ApiKey-v1 YOUR_API_KEY' \
--header 'Content-Type: application/json' \
--data-raw '{ 
  "customerSession": { 
    "profileId": "user-id-456",
    "cartItems": [ 
      { "name": "Product A", "sku": "SKU001", "quantity": 1, "price": 99.99 }
    ],
    "couponCodes": ["WELCOME10"]
  },
  "responseContent": ["customerSession", "effects"]
}'

Management API

This is a RESTful API for back-office operations. It allows you to programmatically do most of what the Campaign Manager UI allows, such as creating campaigns, generating coupons, or managing attributes. It is not designed for the high-load traffic of the Integration API.

05 Data Model

Flexible & Extensible Data Structures

Talon.One's data model is designed to be flexible, allowing you to map any business data point as an attribute for rule building.

Customer Profile

Represents your customer. It has a unique profileId (which should match your internal customer ID) and can hold any number of custom attributes (e.g., tier: 'Gold', lifetimeValue: 5000). Profiles are updated via the Integration API or synced from Segment.

Customer Session

A container for customer activity, typically mapping to a shopping cart or order. It contains cart items, applied coupons, and other session-specific data. The session is the primary object evaluated by the Rule Engine.

Catalog Data

While Talon.One can work with product data sent in the session, you can also sync your product catalog. This allows for more powerful rules based on product attributes (e.g., `brand: 'Nike'`, `margin > 40%`) without sending this data in every API call.

06 Security & Compliance

Enterprise-Grade Security by Design

Security is paramount. Our platform is designed with enterprise-grade features to protect your data and ensure reliable service.

FeatureDescription
Authentication All API requests are authenticated via scoped API keys. Keys should be stored securely on your backend and never exposed client-side.
Rate Limiting Our Third-party APIs (for Segment/Braze) have default rate limits of 60-90 req/sec, which can be increased. The core Integration API is built for high throughput to handle enterprise scale.
PII Handling Talon.One is designed to work without PII. We recommend using non-personally identifiable IDs for customer profiles. We are GDPR compliant and SOC 2 Type II certified.
Infrastructure We offer multi-region hosting options (AWS) to meet data residency requirements and ensure low latency.
