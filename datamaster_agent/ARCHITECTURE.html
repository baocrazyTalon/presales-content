Talon.One Presales Agent
— How it works

A complete reference for the multi-agent AI system: RAG design, system topology, LangGraph state machine, PostgreSQL schema, and the presales web application.

Stack: Python · LangGraph · pgvector · FastAPI
LLM: Gemini 2.5 Pro
Embeddings: gemini-embedding-001 (768d)
DB: PostgreSQL 17 + pgvector 0.8.2
Deploy: GitHub + Vercel
01 — Purpose

What this system does

The Talon.One Presales Agent automates the creation of Technical Presales presentations. A presales engineer describes a prospect (company, industry, use-case, competitor) and the system produces a publication-ready, web-hosted HTML document — in the same quality as the My Muscle Chef deep-dive deck — without writing a single line of HTML.

🧠

AI Orchestration

LangGraph stateful graph routes tasks across five specialist agents: Coordinator, User Preference, Value Selling Strategist, Data Architect, and Document Engineer.

🗄️

Knowledge Memory

pgvector stores embedded Talon.One docs, battle-cards, and prior presentations. The system remembers every client run.

📄

HTML Generator

Outputs responsive HTML/Tailwind with Mermaid.js integration diagrams — matching the mmcTalonone.html visual standard.

🚀

Auto-Deploy

Commits the HTML to GitHub and triggers a Vercel deployment. Every doc gets a public URL within seconds.

Golden rule: This document is updated every time the codebase changes. If you open it and something looks different from what you set up, the document is the source of truth — raise it for review.

02 — Knowledge Architecture

RAG Design — Retrieval-Augmented Generation

RAG is the technique that makes the agent knowledgeable rather than just articulate. Instead of relying on what Gemini already knows, we first retrieve the most relevant Talon.One documentation, battle-cards, and past presentations from a local vector database — then feed that grounded context to the LLM before asking it to write.

Why RAG? LLMs hallucinate product details. RAG grounds every claim in real Talon.One documentation — the same docs the sales team uses. No invented API endpoints, no wrong pricing.

The 7-Stage RAG Pipeline

1
Ingest
PDF · PPTX · XLSX · Images · Web docs loaded into the system
›
2
Chunk
Documents split into overlapping ~500-token passages
›
3
Embed
Gemini gemini-embedding-001 → 768-dim float vector
›
4
Store
Vector + metadata written to documents table in pgvector
›
5
Multi-Query
LLM generates 3–5 diverse query variants per task (technical, business, competitive angles)
›
6
Retrieve
HNSW cosine ANN search per query variant → top-(k×2) candidates each
›
7
RRF Rerank
Reciprocal Rank Fusion merges all result lists → top-k fused chunks
›
8
Generate
Up to 100K chars of context injected into Gemini’s 1M-token window

RAG Evolution Roadmap

The system is designed to evolve. Each version improves retrieval quality:

Version RAG Strategy What it solves Status
v0.1 Naive top-k cosine similarity — single query per task Baseline retrieval: good for well-defined topics ✅ Implemented
v0.2 Multi-query RAG — LLM generates 3–5 rephrased queries per task, results fused via RRF Reduces vocabulary mismatch between user question and doc language ✅ Implemented (v0.7)
v0.3 Reciprocal Rank Fusion (RRF) — rank-based score fusion across all query variants replaces naive dedup Better recall than any single query; no cross-encoder model needed ✅ Implemented (v0.7)
v0.4 Embedding cache — LRU-cached embeddings avoid duplicate Gemini API calls on retries Faster iteration loops, lower API cost ✅ Implemented (v0.7)
v0.5 Contextual compression — retrieved chunks are summarised per-query before injection, reducing prompt token usage Fits more sources into the 128k context window Planned
v1.0 Agentic RAG — the Data Architect decides autonomously when to run more queries, when to crawl Google Drive, and when it has "enough" Self-directed research loop, no fixed iteration count Future

Embedding Model Choice

gemini-embedding-001 (768 dims) — chosen to keep the entire stack on a single provider (Google). No OpenAI key required. Embeddings are stored as vector(768) in PostgreSQL with output_dimensionality=768 (native output is 3072d but HNSW max index dimension is 2000). If you ever switch providers, the entire documents table must be re-embedded and the vector column dimension updated in the schema.

Sources ingested into the knowledge base

📖

Talon.One Docs

Integration API (Session V2), Management API, campaign rules, loyalty concepts, webhooks — crawled via ingest_from_url().

🗂️

Google Drive Assets

PPTX battle-cards, XLSX pricing sheets, PDF one-pagers — indexed from a shared Drive folder defined by GOOGLE_DRIVE_ROOT_FOLDER_ID.

📓

Notion Workspace

Internal Talon.One knowledge pages — product wikis, battle-cards, onboarding notes. Searched by topic via the Notion API, selectively ingested into pgvector. Controlled by NOTION_API_KEY and NOTION_MAX_PAGES. Chunked at 800 words with 80-word overlap for optimal retrieval granularity.

🌐

Web Scraping

Competitor API docs, prospect loyalty/promotions pages, any public URL. Scraped via tools/web_scraper.py — respects robots.txt, rate-limited, chunked at 800 words. Modes: single page (scrape_and_ingest), sitemap parse (scrape_sitemap), or link-following crawl (crawl_links). Accessible via CLI + POST /api/scrape and POST /api/crawl API endpoints.

🖼️

Architecture Images

PNG/JPEG diagrams described by Gemini 2.5 Pro (native multimodal), then the description is embedded and stored — so visual content is searchable.

📝

Past Presentations

Every generated HTML doc is ingested back into the store. Future runs can retrieve successfully used sections and patterns.

04 — Orchestration

LangGraph State Machine + Persistent Memory

LangGraph is used to define a stateful, cyclic graph where each node is an agent function that reads from and writes to a shared AgentState TypedDict. Edges can be conditional — the Coordinator validates the output and either approves it for deployment or loops back for another retrieval + generation cycle.

Three memory layers — all backed by PostgreSQL:
LayerLocationWhat is storedScope
Short-termAgentState.messagesFull conversation turns, auto-merged by add_messagesSingle run
Workinglanggraph_checkpoints tableFull AgentState snapshot after every node. Resume after crash with --resumePer prospect thread
Long-term / Semanticdocuments table (pgvector) + user_insights table768-dim embeddings of document chunks; personal coaching insights are stored in sales_playbook namespace and used for dynamic context injectionGlobal, permanent

Thread model: each prospect gets a stable thread_id derived from their company slug (e.g. "My Muscle Chef" → "my-muscle-chef"). Revisiting the same prospect resumes from the last checkpoint via python main.py --resume. History is viewable via python main.py --history <company>.

05 — Agent Definitions

Five Specialist Agents

🎯 Coordinator — agents/coordinator.py

Role: Project Manager. Decomposes the user brief into an ordered task list. After the document is generated, validates it against the 7-point best-practices checklist and routes the graph accordingly.

Phase A (first call): sends a task decomposition prompt to Gemini 2.5 Pro → returns JSON {"tasks": [...]}.
Phase B (after HTML exists): runs _validate_html() → sets validation.passed.
Temperature: 0.2 (low — structured output needed).

🗄️ Data Architect — agents/data_architect.py

🧠 User Preference — agents/user_preference.py

Role: Active learning handoff. Listens for explicit user feedback via state.user_feedback and persists it to the sales_playbook namespace (Postgres user_insights table). Ensures the agent learns from coaching signals across independent sessions and thread IDs.

Function: - Captures statements like "Focus more on the Loyalty API for Tier 1 clients" - Calls record_user_preference() in core/store.py - Clears state.user_feedback after persistence - Sends control back to Value Selling node for strategic re-analysis

Role: Knowledge Base. Converts the task list into semantic queries, runs them against pgvector, pulls assets from Google Drive, fetches relevant Notion pages, scrapes web URLs provided in the prospect intake (competitor docs, prospect loyalty pages), and uses Gemini 2.5 Pro (natively multimodal) to describe images — no separate Vision API call needed. Returns a deduplicated list of RAGChunk objects.

Dedup strategy: chunks are keyed by source_id + content[:60] — no identical passage appears twice even across multiple queries.
Cap: max 5 Google Drive files · max NOTION_MAX_PAGES (default 5) Notion pages · max WEB_SCRAPER_MAX_PAGES (default 20) web pages per run (configurable).
Notion flow: search → extract full page text → surface first 2 000 chars as an in-session chunk → background-persist all chunks to pgvector for future runs.
Web scraping flow: for each URL in prospect.urls_to_scrape: fetch page → BeautifulSoup extraction → 2 000-char preview as in-session chunk → full content chunked & persisted to pgvector. Respects robots.txt, 1s delay between requests.
Temperature: 0.0 (deterministic retrieval decisions).

✍️ Document Engineer — agents/document_engineer.py

Role: Full-Stack Developer. Receives the retrieved chunks and generates a self-contained HTML/Tailwind/Mermaid document. If called on a retry (validation failed), previous failure notes are injected into the prompt for self-correction. Uses Gemini 2.5 Pro's large 1M-token context window — the entire knowledge base can be passed in a single call if needed.

Context window budget: max 12,000 characters of RAG context injected per run (well within Gemini's 1M limit — can be expanded).
Agenda-driven: If the Value Selling agent produced an agenda, the Doc Engineer follows that exact structure — sections, talking points, and narrative arc are injected into the prompt.
Brand colours: Talon.One defaults (#FF6900, #1A1A2E, #00C2CB) unless prospect assets override.
Temperature: 0.4 (some creativity for narrative sections).

💡 Value Selling Proposition — agents/value_selling.py

Role: Strategic Storyteller. Analyses the prospect context and retrieved knowledge to craft a persuasive sales narrative. Produces a recommended document agenda (table of contents), must-have talking points, a narrative arc, and competitor kill-shots. Sits between Data Architect and Document Engineer in the graph — it tells the Doc Engineer what to write and how to frame it.

Narrative framework: Problem → Insight → Solution → Proof → Call-to-Action.
Outputs: selling_agenda (ordered sections with talking points), narrative_arc, key_talking_points (categorised by integration / competitive_advantage / customer_proof / technical_differentiator / business_value), competitor_kill_shots.
Industry awareness: Adapts angles for Retail, QSR, iGaming, Subscription, Travel verticals.
Max sections: 8 — quality over quantity.
Temperature: 0.3 (creative but still structured).

06 — Data Layer

PostgreSQL 17 + pgvector 0.8.2 Schema

Three tables. Spun up automatically via init.sql when Docker Compose starts. Managed locally — no cloud database required to get started.

PostgreSQL usage overview

This system uses PostgreSQL for:

  • pgvector document semantics: all ingested knowledge chunks in documents (RAG queries)
  • presentation audit logs: presentations stores generated slides / HTML outputs
  • graph checkpointing: LangGraph runs persisted in langgraph_checkpoints, langgraph_blobs, langgraph_writes by PostgresSaver
  • user preference memory: user_insights (new sales_playbook namespace, active learning)

Connection string: postgresql+psycopg://presales:presales_secret@localhost:5432/presales_db — override via DATABASE_URL in .env. The +psycopg suffix is required by LangChain-Postgres; the raw psycopg3 connections in tools/postgres_rag.py use postgresql:// (no suffix).

09 — Getting Started

Step 1 — Local Environment Setup

Before running any agent, you need Python 3.11+, Docker, and your API keys. Follow these steps in order:

1a — Create your .env file

# In presales_agent/ cp .env.example .env # Then open .env and fill in: GOOGLE_API_KEY=AIza... # Gemini API key from Google AI Studio LLM_MODEL=gemini-2.5-pro EMBEDDING_MODEL=gemini-embedding-001 GITHUB_TOKEN=ghp_... GITHUB_REPO=your-org/presales-content VERCEL_TOKEN=... VERCEL_PROJECT_ID=...

1b — Start the database

# From presales_agent/ directory: docker compose up -d # Verify it's healthy: docker compose ps

1c — Install Python dependencies

python -m venv .venv source .venv/bin/activate # Windows: .venv\Scripts\activate pip install -r requirements.txt

1d — Ingest your first documents

# Quick-start: index the key Talon.One doc pages python - <<'EOF' from tools.postgres_rag import ingest_from_url pages = [ "https://docs.talon.one/docs/dev/integration-api/session-v2", "https://docs.talon.one/docs/product/account/managing-applications", "https://docs.talon.one/docs/dev/integration-api/overview", ] for url in pages: ingest_from_url(url) print(f"✅ Ingested: {url}") EOF

1e — Run your first agent job

python main.py # Interactive prompts will ask for company name, industry, etc. # Or pass a JSON file: python main.py --prospect examples/dali.json
